## 第一周：最小可用版本（MVP）

**一句话目标：** 上传一份 PDF，提问，系统从文档中找到相关内容并回答，标出引用来源。

### Day 2: 文本分块

**目标：** 把长文本切成固定大小的 chunk，并且相邻 chunk 之间有 overlap。

**代码文件：** `src/chunker.py`

**要写的函数：**
```python
def chunk_by_size(pages: list[dict], chunk_size: int = 512, overlap: int = 128) -> list[dict]:
    """
    输入: Day1 的页面列表
    输出: [{"chunk_id": 0, "text": "第一个chunk...", "source_page": 1}, 
           {"chunk_id": 1, "text": "第二个chunk...", "source_page": 1}, ...]
    
    chunk_size: 每个chunk最多512个字符
    overlap: 相邻chunk之间重叠128个字符
    """
    pass
```

**关键细节：**
- 跨页处理：如果一页文本在分块结尾还剩不到 chunk_size 的一半，不要强行切，跟下一页的开头拼在一起
- overlap 的目的：防止一个重要句子正好被切断在两个 chunk 之间，检索时两边都搜不到
- chunk_id 必须唯一且递增 → 后面查 Chroma 要靠它定位原文

**为什么 512 和 128：** 512 字符大约是中文 300-400 字，是一段完整的论述长度。128 字符 overlap 保证被切断的句子在相邻 chunk 里至少完整出现一次。第二周你会做对比实验来验证这两个数字。

**验证方法：** 带入 Day1 的结果，打印前 3 个 chunk，确认文本连续且无截断，source_page 正确。

**与后续关系：** chunk 列表是 Day 3 embedding 的输入。chunk 质量决定了检索质量的上限。

---

### Day 3: Embedding 向量化

**目标：** 把每个 chunk 转成一个固定维度的浮点数向量，语义相近的 chunk 向量之间距离相近。

**代码文件：** `src/embedder.py`

**要写的函数：**
```python
class Embedder:
    def __init__(self, model_name: str = "BAAI/bge-small-zh-v1.5"):
        """加载本地 embedding 模型"""
        pass
    
    def embed_chunks(self, chunks: list[dict]) -> list[dict]:
        """
        输入: Day2 的 chunk 列表
        输出: 每个 chunk 增加一个 "embedding" 字段，值为 List[float]
        """
        pass
```

**为什么选 BGE-small：**
- 中文效果好（BAAI 是智源研究院的模型）
- small 版本只有 24MB，CPU 上跑也很快，不占显存
- 向量维度 512，后续检索计算量小
- 换成 BGE-large 只需要改一行 model_name

**关键概念你要理解（面试会问）：**
- Embedding 到底是什么：把"语义"压缩成数字向量。"苹果很好吃"和"苹果手机很好用"两个 embedding 向量之间的距离应该远，但 embedding 自己看不懂这两个"苹果"有什么不同——它只知道上下文不一样
- 为什么不用 OpenAI 的 text-embedding-3：收费 + 网络延迟 + 数据离开本地。本地模型零成本无限调用

**验证方法：** 找两个意思相似的句子和两个意思不同的句子，分别计算 embedding 的余弦相似度，确认相似的 > 0.8，不相似的 < 0.4。

**与后续关系：** embedding 向量是 Day 4 Chroma 存储和检索的基础。没有向量就没有语义检索。

### Day 4: 向量存储与检索

**目标：** 把所有 chunk 的向量存进 Chroma，实现"用户提问 → 向量化 → 找最相似的 k 个 chunk → 返回原文"。

**代码文件：** `src/retriever.py`

**要写的函数：**

```python
               2. 在 Chroma 中做 cosine 相似度检索
               3. 返回 top_k 个最相似的 chunk
        输出: [{"chunk_id": 3, "text": "...", "score": 0.92, "source_page": 2}, ...]
        """
        pass
```

**关键细节：**
- Chroma 的 `collection.query()` 默认返回 cosine 距离。确认这个默认行为，理解为什么 cosine 比欧氏距离更适合文本相似度
- metadata 里一定要存 source_page 和原始 text → 否则检索回来一堆向量，不知道原文是什么，更不知道来自第几页
- 去重：如果同一个 chunk 的相邻 chunk 也被检索到了（因为 overlap 导致内容重复），保留得分高的

**验证方法：** 上传 PDF，建好索引后，问一个能明确在文档中找到答案的问题，看 top-5 结果里是否包含正确答案。记录命中率。

**与后续关系：** retriever 是整个系统的"引擎"。Day 5 的 prompt 构建依赖它返回的相关 chunk。第二周的重排序也插在它返回结果之后。

## 第二周：检索质量提升

**一句话目标：** 把检索准确率从"有时候能查到"提升到"基本每次都能查到"。

**与第一周的关系：** 第一周的固定分块和纯向量检索是最简单的基线。这一周的每一项改进都是在第一周的基线之上做 A/B 对比，每一项都有数据证明"为什么改了更好"。

---

### Day 8-9: 语义分块

**目标：** 替代 Day2 的固定长度切分，改为基于语义边界切分。

**原理：** 固定 512 字符不管内容，可能在句子中间切断。语义分块的做法是：算相邻句子的 embedding 相似度，相似度突然降低的地方就是"话题转折点"，在那里切。

**算法步骤：**
1. 把文本按句号/换行拆成句子
2. 给每个句子算 embedding
3. 算相邻句子的余弦相似度
4. 如果连续 N 个句子之间的相似度都低于阈值 → 这是一个语义边界
5. 在边界处切分，确保每个 chunk 不超过 max_chars

**验证方法：** 用同一个 PDF，分别用 Day2 的固定分块和 Day9 的语义分块建两个索引，问同样 5 个问题，对比检索命中率。记录在表格里，这就是第二周结束时要展示的"对比实验数据"。

---

### Day 10-11: 混合检索

**目标：** 在向量检索的基础上叠加关键词检索，互补短板。

**原理：**
- 向量检索擅长"意思相近但用词不同"的匹配。例如用户问"深度学习如何训练"，文档里写的是"神经网络通过反向传播优化参数"→ 向量能匹配上
- 关键词检索（BM25）擅长专有名词、数字、代码等精确匹配。例如用户问"第3.2节提到了什么"，向量不擅长"第3.2节"这种精确匹配 → BM25 擅长
- 混合 = 两种检索都跑，然后加权融合排序

**实现步骤：**
1. 用 `rank_bm25` 库实现关键词检索
2. 向量检索保持不变
3. 融合策略：RRF（Reciprocal Rank Fusion），不用调权重的简单融合算法

**不需要自己实现 BM25 算法。** 用 `rank_bm25` 库调用即可。你要理解的是"什么时候关键词检索比向量检索好"，不是"BM25 的数学推导"。

---

### Day 12-13: 重排序

**目标：** 初检返回 top-k（比如 20 条），用更精准但更慢的模型对这 20 条重新排序，返回前 5 条。

**原理：**
- 向量检索用的是 embedding（很快但不够精细）
- Cross-encoder 把问题和候选文档拼接在一起，作为一个分类问题算相关性分数（慢但更准）
- 所以策略是：初检用 embedding 快速召回一批候选 → cross-encoder 精排 → 返回 top 几条

**为什么要这一步：** embedding 检索返回的 top-5 里，真正的第 2 名可能排在第 7 位。reranker 把它提到前 5，LLM 做生成的时候就能看到更相关的上下文。

**用 sentence-transformers 的 CrossEncoder 类，不需要自己训练模型。**

## 第三周：多轮对话记忆系统

**一句话目标：** 用户能跟系统"聊天"，追问、澄清、切换话题，系统记得前面聊了什么。

### Day 15-16: 对话记忆数据结构

**目标：** 设计并实现记忆存储结构。

**代码文件：** `src/memory.py`

**数据结构设计：**
```python
class ConversationMemory:
    def __init__(self, max_turns: int = 10):
        self.turns = []        # [{"role": "user", "content": "..."}, 
                               #  {"role": "assistant", "content": "...", "sources": [...]}]
        self.max_turns = 10    # 最多保留最近10轮（防止prompt太长）
        self.summary = ""      # 旧对话的摘要（10轮之前的压缩到这里）
    
    def add_turn(self, user_msg: str, assistant_msg: str, sources: list):
        """添加一轮对话"""
        pass
    
    def get_context(self) -> str:
        """把记忆格式化成可注入prompt的文本"""
        pass
```

**关键设计决策：**
- max_turns=10：保留最近 10 轮完整对话。超过 10 轮的压缩成一句摘要存到 self.summary
- 摘要策略：用 LLM 对旧对话做一次总结，只保留"用户在了解什么主题"这个高层信息。这是你 TTT 研究中对"记忆压缩"的理解的直接应用

**验证方法：** 模拟 15 轮对话，检查 memory 对象是否正确保留最近 10 轮，前 5 轮是否被压缩成摘要。

### Day 17-18: 记忆注入 Prompt

**目标：** 改造 Day5 的 prompt_builder.py，把记忆注入到 prompt 中。

**改造点：** 在 prompt 模板里新增一个"对话历史"区段：

```
系统指令: ...

对话历史（最近几轮）:
用户: 什么是RAG？
助手: RAG是检索增强生成...

用户: 那它的主要组件有哪些？
助手: 主要包括检索器和生成器...

文档上下文: (Day4检索到的chunk)
...

用户当前问题: 第一个组件具体怎么实现？
```

**关键点：** 用户问"第一个组件"时，LLM 看到对话历史里的"检索器和生成器"，就知道"第一个"指的是"检索器"。这解决了指代消解问题。

**与 Day 5 的 prompt_builder 的区别：** 只是 build_rag_prompt 函数多接收一个参数 `conversation_history: str`。函数签名变了，里面加了一块文本拼接。核心逻辑完全不变。

### Day 19-20: 对话重写

**目标：** 处理用户模糊提问。用户说"那第二个呢？"，系统先把这句话重写成"那RAG的第二个主要组件（生成器）具体怎么实现？"，再做检索。

**为什么需要这一步：** 用户原始的"那第二个呢"拿去检索，向量相似度匹配不到任何有用内容。必须先补全。

**实现：** 在检索之前加一步——用 LLM（同一个 API，不同的 system prompt）把模糊提问重写成完整问题。prompt 里给它上一轮对话作为上下文。

```
system: 请根据对话历史，把用户的模糊提问重写成一个完整、独立的问题。
history: 用户刚问了"RAG的主要组件有哪些"，助手回答"检索器和生成器"
当前模糊问题: 那第二个呢？
重写后: RAG的第二个主要组件（生成器）具体怎么实现？
```

**验证方法：** 测 5 种常见模糊提问模式：
- "那第一个呢？" → 补全了指代
- "展开说说" → 补全了想展开的主题
- "还有吗？" → 追问更多
- "不对，我说的是XX" → 纠正理解
- 直接切话题：问了一个跟前面完全无关的问题 → 不做改写

### Day 21: 多轮对话端到端测试

**目标：** 用一组多轮对话测试用例，验证整个记忆系统。

**测试场景设计（5 组，每组 5-8 轮）：**
1. **连续追问**：问 A→追问细节→再追问→再追问
2. **话题切换**：聊 A→切到 B→切到 C→回到 A（验证记忆是否保留）
3. **指代消解**：问题含"它""这个""前面提到的"
4. **纠正与澄清**：用户说"不对，我说的是..." 系统要更新理解
5. **边界压力**：30 轮对话后，系统行为和记忆状态是否正常

每组测试记录：预期结果 vs 实际结果。这份测试报告加到 README 里。
